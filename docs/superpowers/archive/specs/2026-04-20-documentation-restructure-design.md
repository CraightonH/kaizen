# Design: Documentation Restructure + Maintenance Workflow

Generated by /brainstorming on 2026-04-20
Branch: master
Repo: CraightonH/kaizen
Status: APPROVED

## Problem Statement

The kaizen docs have drifted significantly behind the codebase. Two pain points
dominate: factual drift (docs reference things that no longer exist or work
differently) and coverage gaps (new features ship with no doc at all). The
project is moving fast enough that per-PR doc maintenance is impractical;
the update cadence must align with feature completion, not individual commits.

The primary audiences are:
1. **Core contributors** — need accurate internals docs to work on kaizen itself
2. **Plugin/marketplace authors** — need complete authoring guides to build
   against kaizen's APIs without reading source

This repo documents kaizen core only. Plugins live in a separate repo and are
out of scope.

## Docs Structure

Replace the current flat `docs/` with a three-layer hierarchy:

```
docs/
  README.md                    ← journey checklist + index/map
  concepts/
    platform.md                ← why kaizen exists, what it is (replaces DESIGN.md)
    architecture.md            ← kernel model, event bus, registry
    plugin-model.md            ← merged plugin-api + plugin-loading (conceptual layer)
    security.md                ← plugin security model
  guides/
    plugin-authoring.md        ← primary guide: scaffold → implement → test → validate
    marketplace-authoring.md   ← write and publish a marketplace
    contributing.md            ← contribute to kaizen core
  reference/
    plugin-api.md              ← types, manifest schema, API surface
    plugin-standards.md        ← required vs guideline rules
    host-api.md                ← host API that plugins call into
    plugin-secrets.md          ← secret provider interface
  superpowers/                 ← dev artifacts (specs/plans), not living docs
```

`docs/README.md` is the most important file in the restructure. It serves two
purposes simultaneously:
1. **Navigation index** — every doc linked with a one-line description
2. **Plugin author journey checklist** — ordered list of questions a plugin
   author must be able to answer, each linked to the doc that answers it

The journey checklist makes coverage gaps visible: any unanswered step without
a link is a documented gap, not a silent omission.

### Plugin Author Journey Checklist

The following questions must all have answers in `docs/`:

1. What is a plugin and what can it do?
2. How do I scaffold a new plugin?
3. How do I register tools?
4. How do I use the host API (secrets, config, event bus)?
5. How do I declare capabilities and dependencies?
6. How do I test my plugin locally?
7. How do I validate it (`kaizen plugin validate`)?
8. How do I publish to a marketplace?

## Migration Plan (One-Time)

### Delete
- `DESIGN.md` — retire. Its "why/what" essence migrates to `docs/concepts/platform.md`.
  Session annotations, deprecated role language, and approval history are discarded.
- `docs/adversarial-review.md` — brainstorm artifact, not living documentation.
- `docs/plugin-migration-capability-registry.md` — migration guide for a completed
  migration. Discard once confirmed no active migration is in progress.

### Merge / Restructure
- `docs/plugin-api.md` + `docs/plugin-loading.md` →
  `docs/concepts/plugin-model.md` (conceptual) + `docs/reference/plugin-api.md` (reference)

- `CONTRIBUTING.md` → `docs/guides/contributing.md`; root `CONTRIBUTING.md`

## Maintenance Workflow

### Trigger
Doc updates run at **feature completion** — when the last PR of a feature chain
ships — not per individual PR. This aligns with the natural breakpoint where the
feature surface is stable enough to document accurately.

### Mechanism: `kaizen:update-docs` skill

A new project-local skill that:
1. Reads `git diff master...HEAD` to understand what changed in the branch
2. Scans `docs/` for files touching affected areas
3. Flags stale claims, missing coverage, and outdated examples
4. Patches what it can automatically; surfaces what needs human judgment
5. Verifies every plugin author journey step still has a valid answer
6. Commits doc changes as a `docs(...)` commit before the branch ships

This is a standalone skill (not a wrapper over any other skill). It knows:
- The repo audience (plugin/marketplace authors, core contributors)
- The doc structure (which layer each change touches)
- To skip `docs/superpowers/` (dev artifacts, not living docs)
- The journey checklist (coverage check, not just staleness check)

### Enforcement

Add to kaizen's `CLAUDE.md`:
> Before invoking `superpowers:finishing-a-development-branch`, run `kaizen:update-docs`. This is mandatory, not optional.

Agents see this at session start. It becomes a workflow contract.

### When `kaizen:update-docs` is *not* needed
- Chore/fix PRs that don't change behavior or API surface
- Internal refactors with no externally visible change
- The skill itself should detect this and exit early with a short report

## What This Does Not Cover

- A static site generator (VitePress, Docusaurus) — intentionally deferred.
  Adopt when the first external plugin author arrives or at v1.0, whichever comes first.
- TSDoc comments in kaizen source — additive, not part of this effort. Future
  standard: plugin authors should be quasi-required to document their public API
  surface with TSDoc. The plugin scaffolder should generate TSDoc boilerplate on
  `kaizen plugin create` so the pattern is established from the start.
- `docs/superpowers/` organization — these are session artifacts and are out of scope.
