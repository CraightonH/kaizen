## Problem Statement

LLM coding agents are locked to single providers and extend only via MCP — which
requires writing a full server process. The prior design solved "any CLI as an LLM
tool with zero config." This design expands the scope: kaizen is a platform for
building, sharing, and composing LLM harnesses.

A harness is a combination of tools (what the LLM can call) and lifecycle behavior
(how the session runs). Simple harnesses wrap a few CLIs. Complex harnesses add custom
tools, mutate LLM behavior, and encode domain knowledge. All of them are built with the
same plugin primitives and shareable with one command.

The "Godot harness" example is the north star: someone publishes a kaizen harness for
Godot game development — it registers Godot CLI tools, sets up project-specific context,
maybe hooks into tool results to post-process output. Another developer installs it with
one command and has a domain-specific LLM assistant in minutes.

## What Makes This Cool

The unit of sharing is the harness, not the config file. A harness is an npm package.
You install it, and kaizen wires everything — tools, lifecycle hooks, session config —
automatically. The developer who built the Godot harness encodes their workflow once.
Everyone else benefits.

This is VS Code extensions for LLM assistants, without the IDE.

## Constraints

- Language: TypeScript. Runtime: Bun. Ships as a standalone binary via GitHub Releases
  (`bun build --compile`). No Node.js or Bun required on the user's machine.
  Plugin ecosystem is still npm — Bun's module resolution is Node-compatible.
  `npm install -g kaizen` also works as a secondary install path (for Node users).
- Open source (MIT).
- Zero config from the user's perspective for the default use case.
- Core is minimal: plugin loader + event bus + LLM/tool primitives. No session loop,
  no UI, no tools — all of these are plugins. Core emits exactly one internal event
  (`__core:ready`) and exposes raw LLM and tool execution primitives to plugins.
- MVP default stack (three built-in plugins, ships with kaizen):
    kaizen-plugin-lifecycle  — session loop
    kaizen-plugin-ui-terminal — terminal I/O (display + input)
    kaizen-plugin-cli         — CLI tool introspection
  With these three you get a working chat interface and a tool to build more tools.
- Backward compatibility: the Phase 1 `kaizen add <cli>` / `kaizen run` UX is preserved.
  Powered by the three built-in plugins, transparent to the user.
- Security: destructive command guard is implemented in `kaizen-plugin-cli`.
  Core has no opinion on tool safety.
- Phase 2 scope: plugin system, lifecycle, shareable harnesses, npm keyword-based
  discovery. Phase 3: dedicated registry UI with ratings and install counts.

## Definitions

**Role:** A named capability that a plugin may provide (e.g. `'lifecycle'`, `'ui'`).
Roles decouple plugins from specific implementations — `depends: ['lifecycle']` works
with any lifecycle plugin. Core enforces: if any loaded plugin declares a role in its
`depends[]`, exactly one loaded plugin must `provide[]` that role. Roles with no
dependents are not enforced.

**Plugin:** A self-contained npm package that exports a `KaizenPlugin` default.

**Harness:** A npm package that is both a plugin AND ships a `kaizen.json` with
recommended config. A harness may register custom tools, or simply compose and
configure existing plugins — custom tools are optional.

## Premises

1. A plugin is the right unit of extensibility. Not a config file, not a CLI wrapper —
   a self-contained npm package that can be installed, removed, and shared.

2. Lifecycle hooks are the right primitive for plugin power. Plugins hook into named
   events rather than composing tools through a functional pipeline.

3. "Shareable harness" means installable in one command — not a JSON config to copy-paste.
   A harness is an npm package that pulls in its own plugin dependencies.

4. (KEY REVISION) Core = plugin loader + event bus + LLM/tool primitives only. The
   session loop, UI, and CLI introspection are all plugins. Nothing in core defines
   what a "session" looks like, how output is displayed, or what tools exist. Someone
   building a fully autonomous harness or a web UI replaces the relevant plugin — core
   never changes.

5. npm is the plugin distribution channel. Plugins declare capability roles
   (`provides: ['lifecycle']`, `provides: ['ui']`). Other plugins depend on roles, not
   specific names — `depends: ['lifecycle']` works with any lifecycle plugin, enabling
   drop-in replacement. Core enforces: exactly one plugin per role that requires it.
   Discovery is npm keyword-based in Phase 2. Dedicated registry is Phase 3.

## Plugin System Architecture

### Core Responsibilities

Core does exactly three things:
1. **Plugin loader** — resolve npm packages, topo-sort by `depends`, call `setup(ctx)`
2. **Event bus** — `defineEvent`, `on`, `emit`. No events are hardcoded in core.
3. **LLM + tool primitives** — raw `llm.send()`, `llm.stream()`, `tools.execute()`,
   `tools.register()`. No session loop. No I/O. Just the primitives.

Core emits one internal event: `__core:ready` (all plugins initialized). The lifecycle
plugin handles it and starts the session. If no lifecycle plugin is present, kaizen
exits: "No lifecycle plugin found. Add one to kaizen.json."

Core has no tools, no display logic, no user input handling.
CLI commands: `kaizen run`, `kaizen init`, `kaizen apply`, `kaizen install`,
`kaizen plugin install/remove/list`, `kaizen add` (delegated to kaizen-plugin-cli).

### Plugin Manifest Interface

```typescript
interface KaizenPlugin {
  name: string;          // kebab-case. must match config namespace key in kaizen.json
  apiVersion: string;    // semver — core warns if major != core's PLUGIN_API_VERSION
  provides?: string[];   // capability roles this plugin fulfills (e.g. 'lifecycle', 'ui')
  depends?: string[];    // capability roles or plugin names required before setup()
  setup(ctx: PluginContext): Promise<void>;
}
// Core enforces: exactly one loaded plugin per role that any plugin depends on.
// Zero providers for a required role → fatal startup error.
// Two providers for the same role → fatal startup error.

// Plugin entry point: default export must be a KaizenPlugin.
// e.g. export default { name: 'my-plugin', apiVersion: '2.0.0', setup }

interface PluginContext {
  // Tool registration (INITIALIZING state only):
  registerTool(tool: ToolDefinition): void;

  // Event bus:
  defineEvent(name: string): void;                          // declare a new event type
  on(event: string, handler: EventHandler): void;           // subscribe (any event, typed or custom)
  emit(event: string, payload?: unknown): Promise<unknown>; // fire event, returns first non-void result

  // Plugin config and logging:
  config: Record<string, unknown>;  // plugin-specific config slice from kaizen.json
  log(msg: string): void;           // structured logger (prefixed with plugin name)

  // Runtime primitives (for lifecycle/orchestration plugins — convention: only the
  // lifecycle plugin should drive the session loop via these):
  runtime: {
    llm: {
      send(messages: Message[], tools: ToolDefinition[]): Promise<LLMResponse>;
      stream(messages: Message[], tools: ToolDefinition[]): AsyncIterable<LLMStreamChunk>;
    };
    tools: {
      list(): ToolDefinition[];
      execute(name: string, args: Record<string, unknown>): Promise<ToolResult>;
    };
  };
}

// If setup() throws, kaizen logs the error and skips the plugin.
// The session continues without it. Core never crashes on plugin init failure.
// Plugin dependency ordering: add a `depends?: string[]` field to KaizenPlugin.
// Core topologically sorts plugins before setup(). Cycles → fatal error at startup.
// If a dependency is missing from kaizen.json, log a warning and continue.

// ToolResult — returned by execute() and passed to tool:after hooks:
interface ToolResult {
  ok: boolean;
  output?: string;       // human-readable string. sent to LLM if data absent.
  data?: unknown;        // structured output (JSON). sent to LLM instead of output
                         // if both present, LLM receives data; output is for humans only.
  error?: string;        // error message. sent to LLM if ok=false.
  exit_code?: number;    // for subprocess-based tools. informational only.
}
// If execute() throws (uncaught exception): core wraps it as
// { ok: false, error: err.message }. tool:after still fires with this result.
// Stack traces are logged to stderr (debug level) but never sent to the LLM.

// Core validates tool arguments against parameters: JsonSchema (via ajv) before calling execute().
// If validation fails: execute() is NOT called; tool:after fires with
// { ok: false, error: 'Invalid arguments: <ajv message>' }.

// Tool naming conflicts: if two plugins register tools with the same name,
// first registration wins. Second is skipped with a warning:
// "Tool <name> already registered by <plugin1>. Skipping duplicate from <plugin2>."

// Tool definition (OpenAI function-calling format from Phase 1 design)
interface ToolDefinition {
  name: string;
  description: string;
  parameters: JsonSchema;
  destructive?: boolean;  // if true, core prints the call and prompts confirmation
                          // (unless session started with --allow-destructive)
  execute(args: Record<string, unknown>): Promise<ToolResult>;
}

// Lifecycle events and their hook semantics:
//
//  session:start  — fires once at session open. Read-only. Cannot abort session.
//  session:end    — fires once at session close. Read-only.
//  tool:before    — fires before execute(). Handlers run serially in plugin init order.
//                   If a handler returns a non-undefined ToolResult, that result is used
//                   and execute() is skipped. Remaining tool:before handlers are NOT
//                   called (execution short-circuits immediately). If all handlers return
//                   undefined/void, execute() runs normally.
//                   tool:after always fires regardless of short-circuit.
//  tool:after     — fires after execute() (or short-circuit). Receives ToolResult.
//                   Handler may return a new ToolResult to replace it. Returning
//                   undefined/void passes result through unchanged. Runs sequentially.
//  response:before — fires before LLM response is yielded to the user. Read-only.
//                   Cannot suppress or mutate the response in Phase 2.
//  response:after  — fires after response is displayed. Read-only.
//
// All handlers are async and run serially (one completes before the next starts).
// Handler order = plugin initialization order (after topological sort by depends[]).
// Plugins with dependencies initialize AFTER their dependencies, so their handlers
// run AFTER their dependencies' handlers. Example: kaizen-plugin-cli depends on
// ['lifecycle'], so cli's tool:before runs after lifecycle's tool:before (if any).
// Concurrency: tool calls within the same session do NOT run concurrently in Phase 2.
// One tool call completes its full before→execute→after cycle before the next starts.
// If a handler throws, the error is logged to stderr and execution continues.
// Hook errors do not abort the session or the tool call.

// Events are NOT defined by core. They are defined by plugins via ctx.defineEvent()
// during setup() (INITIALIZING state). kaizen-plugin-lifecycle defines the default
// event set. Custom plugins may define additional namespaced events.
// ctx.on() may be called for any event name — if the event was never defined by any
// plugin (after INITIALIZING completes), the handler is registered but fires a warning
// on first emit: "Unknown event '<name>' — possible typo or missing plugin dependency."
// No duplicate event definitions: calling ctx.defineEvent() for an already-defined
// event logs a warning and is ignored (first definition wins).

type EventHandler = (payload?: unknown) => Promise<unknown | void>;

// Payload types for the default lifecycle events (exported from kaizen-plugin-lifecycle
// as TypeScript types for type-safe plugin development against the default stack):
interface SessionContext  { sessionId: string; provider: string; config: KaizenConfig }
interface ToolCallContext { tool: string; args: Record<string, unknown>; sessionId: string }
interface ResponseContext { content: string; sessionId: string }
interface LoopContext     { response: LLMResponse; sessionId: string }

```

### Plugin Loader Contract

```
Resolution: use require.resolve() with paths option:
  const globalRoot = execSync('npm root -g').toString().trim()  // cross-platform
  require.resolve(pluginName, { paths: [
    globalRoot,                   // global npm (npm root -g, works on Win/WSL/Mac/Linux)
    process.cwd() + '/node_modules',  // project-local
  ]})
If pluginName starts with './' or '/' or is an absolute path: treat as local file,
require.resolve() directly. This supports private/local plugins.
If resolution fails: log "Cannot find plugin '<name>'. Install it with:
  npm install -g <name> (global) or npm install <name> (local)" and skip.
Module must be loaded (not re-compiled) — respects Node require.cache.

Required shape: module.default must satisfy KaizenPlugin shape check:
  typeof default.name === 'string' && typeof default.setup === 'function'
If shape check fails: log "Plugin <name> does not export a valid KaizenPlugin" and skip.

Initialization sequence:
1. Parse kaizen.json. Exit with error if provider or plugins fields are missing.
2. Resolve all plugin modules (resolution step above).
3. Topologically sort resolved plugins by depends[]. Cycles → fatal startup error.
4. Call plugin.setup(ctx) for each plugin, in sorted order, serially.
   If setup() throws: log the error + stack (debug), skip plugin, continue.
5. After all setup() calls: warn on any kaizen.json keys that no plugin claimed
   (not 'provider', not 'plugins', not a loaded plugin's name).
   e.g. "Unknown config key 'kai-plugin-typo'. No plugin claimed it."
6. Open LLM session with all registered tools.

Session state machine:
  INITIALIZING  → setup() calls running (defineEvent, on, registerTool valid here only)
  READY         → all plugins initialized, core emits __core:ready
  RUNNING       → lifecycle plugin has started the session loop
  CLOSED        → lifecycle plugin has ended the session

registerTool() and defineEvent() and on() are valid in INITIALIZING only.
Calling registerTool() outside setup() throws: "Cannot register tools after initialization."

Version check: if semver major(plugin.apiVersion) != PLUGIN_API_VERSION,
log warning: "Plugin '<name>' apiVersion <X>, core expects <Y>. Loading anyway."
Minor/patch mismatch: load silently.
PLUGIN_API_VERSION is exported from core as a constant.

Role validation (after all plugins loaded):
- For each role in depends[] across all plugins: exactly one provider must exist.
  Zero → fatal: "No plugin provides role '<role>'. Add one to kaizen.json."
  Two+ → fatal: "Multiple plugins provide role '<role>': X, Y. Remove one."
```

### Lifecycle Timing Diagram

This shows the default stack (kaizen-plugin-lifecycle + kaizen-plugin-ui-terminal).
A custom lifecycle plugin can define a completely different flow.

```
kaizen run "<prompt>"
  │
  ├─ INITIALIZING (core)
  │   ├─ parse kaizen.json, validate roles
  │   ├─ resolve + topo-sort plugins by depends[]
  │   ├─ for each plugin: setup(ctx) → defineEvent(), on(), registerTool()
  │   └─ warn on unclaimed config keys
  │
  ├─ READY → core emits __core:ready
  │           ↓ kaizen-plugin-lifecycle handles it:
  │
  └─ RUNNING (driven by kaizen-plugin-lifecycle):
      ├─ emit session:start
      │
      └─ loop:
          ├─ emit session:loop                    ← kaizen-plugin-ui-terminal hooks this,
          │   (returns { type:'continue', prompt } │  prompts user, returns their input)
          │    or { type:'yield' } for user input  │  autonomous plugins can short-circuit
          │    or { type:'end' } to close)         │  with { type:'continue', prompt: ... }
          │
          ├─ runtime.llm.send(messages, tools)
          │
          ├─ for each tool call from LLM:
          │   ├─ validate args (ajv)
          │   ├─ emit tool:before (first non-void return short-circuits execute)
          │   ├─ runtime.tools.execute() (or skip if short-circuited)
          │   └─ emit tool:after (each handler may replace ToolResult)
          │
          ├─ emit response:before              ← kaizen-plugin-ui-terminal renders here
          ├─ emit response:after
          └─ repeat loop
      │
      └─ emit session:end → CLOSED
```

### Built-in Plugins (MVP — all three ship with kaizen)

All three are bundled in the kaizen package (`dependencies`). Users never install them
manually. All three are auto-included by `kaizen init`. They will be extracted to
standalone npm packages in Phase 3 once the plugin API stabilizes.

#### `kaizen-plugin-lifecycle`
```
provides: ['lifecycle']
depends:  []
```
Handles `__core:ready`, drives the session loop, defines and emits the default event
set: `session:start`, `session:end`, `session:loop`, `tool:before`, `tool:after`,
`response:before`, `response:after`. Uses `ctx.runtime.llm` and `ctx.runtime.tools`
to send prompts and execute tool calls. Has no I/O of its own.

I/O delegation: lifecycle emits `session:loop` and awaits a non-void return to get the
next prompt. If no handler returns a value, session ends cleanly (rather than hanging).
This means kaizen-plugin-lifecycle does NOT need to declare `depends: ['ui']` — it
works without a UI plugin. The UI plugin hooks `session:loop` and provides input when
present. An autonomous plugin can do the same, replacing or short-circuiting the UI.

**Two autonomous patterns:**

Pattern A — Light autonomous (add an orchestrator, keep default lifecycle + UI):
```json
{ "plugins": ["kaizen-plugin-lifecycle", "kaizen-plugin-ui-terminal",
              "kaizen-plugin-cli", "kaizen-plugin-autonomous-orchestrator"] }
```
The orchestrator hooks `session:loop` and short-circuits before UI gets input.
Good for: autonomous agents that still want terminal monitoring/logging.

Pattern B — Heavy autonomous (replace lifecycle entirely):
```json
{ "plugins": ["kaizen-plugin-autonomous", "kaizen-plugin-cli"] }
```
`kaizen-plugin-autonomous` provides `['lifecycle']`, defines its own event set,
and drives the loop without `session:loop` at all. No UI, no human input.
Good for: CI/CD agents, fully headless workflows.

Exports its event payload types as TypeScript types (`SessionContext`, `ToolCallContext`,
etc.) so plugin authors building on the default stack get type safety.

#### `kaizen-plugin-ui-terminal`
```
provides: ['ui']
depends:  []
```
Hooks `session:loop`: prints `> ` prompt, reads a line of stdin, returns
`{ type: 'continue', prompt: input }`. On empty input or `/exit`: returns
`{ type: 'end' }`. Hooks `response:before`: streams LLM response to stdout.
Hooks `response:after` for any post-response formatting (e.g. newline flush).

Replacing this plugin with `kaizen-plugin-ui-tui` or `kaizen-plugin-ui-web` changes
how the session looks and where input comes from — nothing else changes.

#### `kaizen-plugin-cli`
```
provides: []
depends:  ['lifecycle']
```
Full Phase 1 CLI introspection engine (unchanged). Registers lazily-loaded tool schemas
from `--help` introspection. Handles `kaizen add <cli>` delegation. Destructive guard
via `tool:before` hook. Hooks lifecycle events — works with any lifecycle plugin that
defines `tool:before` and `tool:after`. If those events aren't defined (e.g., custom
lifecycle omits them), the guard degrades gracefully with a warning.

### Harness Config: `kaizen.json`

Full schema:

```typescript
interface KaizenConfig {
  provider: string;             // required. matches key in ~/.kaizen/config.json
  plugins: string[];            // required. npm package names, load order = array order
  [pluginName: string]: unknown; // plugin config namespace — key must match plugin.name
}
```

Example:

Default (generated by `kaizen init`):
```json
{
  "provider": "anthropic",
  "plugins": [
    "kaizen-plugin-lifecycle",
    "kaizen-plugin-ui-terminal",
    "kaizen-plugin-cli"
  ],
  "kaizen-plugin-cli": {
    "clis": [],
    "allow_destructive": false,
    "subprocess_timeout_ms": 30000
  }
}
```

Godot harness (swaps nothing, extends tools):
```json
{
  "provider": "anthropic",
  "plugins": [
    "kaizen-plugin-lifecycle",
    "kaizen-plugin-ui-terminal",
    "kaizen-plugin-cli",
    "kaizen-harness-godot"
  ],
  "kaizen-plugin-cli": { "clis": ["gdscript", "godot"] },
  "kaizen-harness-godot": { "project_path": "./my-game" }
}
```

Autonomous harness (replaces lifecycle + UI entirely):
```json
{
  "provider": "anthropic",
  "plugins": [
    "kaizen-plugin-autonomous",
    "kaizen-plugin-cli"
  ],
  "kaizen-plugin-autonomous": { "max_steps": 50, "goal": "fix all failing tests" },
  "kaizen-plugin-cli": { "clis": ["gh", "pytest"] }
}
```

Web UI harness (swaps only the UI layer):
```json
{
  "provider": "anthropic",
  "plugins": [
    "kaizen-plugin-lifecycle",
    "kaizen-plugin-ui-web",
    "kaizen-plugin-cli"
  ],
  "kaizen-plugin-ui-web": { "port": 3000 }
}
```

Config precedence (highest to lowest):
1. CLI flags (`--provider`, `--allow-destructive`, `--tools`)
2. Local `kaizen.json` in the current directory
3. `~/.kaizen/kaizen.json` (global default, created by `kaizen init --global`)
4. Built-in defaults

`kaizen.json` in the project root is the harness definition. `kaizen apply` reads it,
installs missing plugins, and runs the initialization sequence (which warns on unclaimed
config keys after all plugins load). Config namespace key must exactly match plugin.name
(kebab-case). Typos in config keys surface as warnings after initialization.

CLI flags override kaizen.json values for the same key. Array values (e.g., `--tools`)
replace, not append — `--tools gh,kubectl` means only those two CLIs are active, even
if kaizen.json lists more.

Missing `provider` or `plugins` fields: `kaizen apply` exits with a validation error.
Reserved key collision: if a plugin's `name` matches a reserved key (`provider`,
`plugins`), core logs an error and skips the plugin: "Plugin name '<name>' collides
with reserved config key. Rename the plugin."

### Global Provider Config: `~/.kaizen/config.json`

Created interactively on first `kaizen run`. Defines available providers by name.
`kaizen.json["provider"]` references one of these names.

```typescript
interface ProviderConfig {
  adapter: 'anthropic' | 'openai' | 'google' | 'mistral'; // which SDK adapter to use
  model: string;         // model ID passed to the adapter
  api_key_env?: string;  // env var name holding the API key (omit for local/keyless)
  api_key?: string;      // literal key (not recommended; use api_key_env instead)
  baseURL?: string;      // override the adapter's default endpoint — enables local LLMs
}

interface GlobalConfig {
  providers: Record<string, ProviderConfig>;
}
```

**Local LLM support is a first-class design goal.** Most local LLM servers (Ollama,
LM Studio, llama.cpp server, Jan, GPT4All) expose an OpenAI-compatible REST API.
Point the `openai` adapter at their local endpoint via `baseURL`:

```json
{
  "providers": {
    "anthropic": {
      "adapter": "anthropic",
      "model": "claude-opus-4-6",
      "api_key_env": "ANTHROPIC_API_KEY"
    },
    "ollama-llama3": {
      "adapter": "openai",
      "model": "llama3.2",
      "baseURL": "http://localhost:11434/v1",
      "api_key": "ollama"
    },
    "lm-studio": {
      "adapter": "openai",
      "model": "local-model",
      "baseURL": "http://localhost:1234/v1",
      "api_key": "lm-studio"
    }
  }
}
```

A local LLM harness is pure config — no custom plugin needed:
```json
{
  "provider": "ollama-llama3",
  "plugins": ["kaizen-plugin-lifecycle", "kaizen-plugin-ui-terminal", "kaizen-plugin-cli"],
  "kaizen-plugin-cli": { "clis": ["git", "gh"] }
}
```

**Implementation requirement:** The `baseURL` field MUST be wired through to the
Vercel AI SDK adapter at startup. The `createOpenAI({ baseURL, apiKey })` call must
use this value, not a hardcoded string. This must be tested against Ollama in CI
before Phase 2 ships — local LLM support that silently fails on `baseURL` is worse
than not advertising it.

### Shareable Harness Format

A harness is an npm package that is BOTH a plugin (exports `KaizenPlugin`) AND ships
a `kaizen.json` with its default config. This is what distinguishes it from a plain
plugin: `kaizen install` handles the config merge; `kaizen plugin install` does not.

```
kaizen-harness-godot/
  package.json          # "keywords": ["kaizen-harness"]. kaizen-plugin-cli in dependencies.
  kaizen.json           # default config this harness recommends
  index.ts              # default export: KaizenPlugin (registers Godot-specific tools)
  README.md
```

When a user runs `kaizen install <name-or-path>`:
- If `<name-or-path>` starts with `./`, `/`, or is an absolute path: treat as local
  directory. Skip npm install. Read its `kaizen.json` directly.
- Otherwise: resolve `kaizen-harness-<name>` from npm registry.
  `npm install -g kaizen-harness-<name>`

Then:
1. Read the harness package's `kaizen.json` as defaults
2. Merge into local `kaizen.json`:
   - `plugins` array: union. Local plugins first, harness plugins appended. No duplicates.
   - Plugin config namespaces (e.g., `"kaizen-plugin-cli": {...}`): deep merge.
     Arrays within a namespace: union, deduplicated (local items first).
     Scalar values: local wins if present and non-null. `null` in local = absent
     (harness default is used). Log: "Local config overrides '<key>'" on collision.
   - All other top-level keys (except `provider`, `plugins`): local wins if present
     and non-null. Harness value fills in absent or null keys.
   - `provider` key: local always wins, no merge.
3. Run `kaizen apply` with merged config

If no local `kaizen.json` exists, create one from the harness defaults.

A plain plugin (`kaizen plugin install <pkg>`) installs the npm package but skips
config merge. User manually adds the plugin name to `kaizen.json["plugins"]`.

Discovery in Phase 2: `npm search kaizen-harness` and `npm search kaizen-plugin`
(npm keyword conventions). A dedicated registry UI is Phase 3.

## Approaches Considered

### Approach A: Fixed lifecycle, plugins extend it
- Core owns the session loop and event set. Plugins subscribe.
- Simpler core, less powerful — can't replace the loop or UI.

### Approach B: Middleware pipeline
- Plugins inject `(ctx, next)` middleware into named stages.
- Flexible but plugin ordering bugs are subtle.

### Approach C: Full kernel — everything is a plugin (CHOSEN)
- Core = loader + event bus + LLM/tool primitives only.
- Lifecycle, UI, and tools are all plugins with defined capability roles.
- Any layer is replaceable without touching core.
- MVP ships three built-in plugins that prove the API before any third party writes one.

## Recommended Approach

**Kernel model (Approach C).** Core is inert without plugins. The three built-in
plugins make it useful. Everything is replaceable.

MVP delivery order:
1. Define `KaizenPlugin` + `PluginContext` interfaces in `src/types/plugin.ts`
2. Build plugin loader with role validation and topo-sort
3. Build event bus (`defineEvent`, `on`, `emit`) and expose `ctx.runtime`
4. Write `kaizen-plugin-lifecycle` — handles `__core:ready`, drives the loop
5. Write `kaizen-plugin-ui-terminal` — terminal I/O via `session:loop` + `response:before`
6. Write `kaizen-plugin-cli` — Phase 1 introspection engine, hooks `tool:before/after`
7. Wire all three, run `kaizen run` end-to-end — this is the MVP
8. Write `kaizen-plugin-noop` (external) — smoke test that the API has no coupling to the built-ins

Step 8 is the quality gate. If noop requires touching `src/types/plugin.ts`: the API
is coupled to the built-in implementations and must be fixed before shipping.

## Open Questions

1. **Plugin isolation:** Phase 2 answer: in-process, clear security disclaimer in docs
   ("plugins have full Node.js access — only install plugins you trust"). Phase 3:
   evaluate `worker_threads` sandboxing.

2. **Multi-value flags in kaizen-plugin-cli:** repeated flags (`--label k=v --label k2=v2`)
   and comma-separated values — schema representation deferred to implementation, will
   surface during Phase 1 test targets (gh, kubectl, doppler).

3. **Provider interface:** `ctx.runtime.llm.send()` and `.stream()` are
   provider-agnostic by design — signatures do not expose the Vercel AI SDK directly.
   Core wraps the SDK in a provider adapter at startup. Phase 2: only the provider
   specified in `kaizen.json["provider"]` is available per session. Phase 3: plugins
   may supply their own provider via a `ctx.runtime.llm.withProvider(adapter)` API.
   Custom plugins needing a different LLM in Phase 2 may instantiate their own SDK
   instance directly, but results are not visible to the main session loop.

4. **Multi-value CLI flags:** repeated flags and comma-separated values — deferred to
   implementation, will surface during Phase 2 test targets (gh, kubectl, doppler).

## Distribution Plan

**kaizen binary:**
- Primary: `bun build --compile --target=bun-linux-x64 ./src/cli.ts --outfile kaizen`
  → self-contained binary, no Node.js or Bun required on user's machine.
  Bun supports cross-compilation via `--target` flag (bun-linux-x64, bun-linux-arm64,
  bun-darwin-x64, bun-darwin-arm64, bun-windows-x64). All five built in CI.
  GitHub Releases: one binary per platform. One-line install: `curl -fsSL https://kaizen.sh/install | sh`
  Note: plugin resolution via `require.resolve()` works from the compiled binary because
  resolution reads from the filesystem (npm root -g, project node_modules) — not from
  the binary's bundle. Verify this in CI with an integration test that loads a plugin
  from a compiled binary before shipping.
- Secondary: `npm install -g kaizen` — runs on Node.js as-is (Bun is Node-compatible).
- Homebrew tap: Phase 3.

**Plugin/harness distribution:**
- Plugins: npm package with keyword `kaizen-plugin`. `npm search kaizen-plugin`.
- Harnesses: npm package with keyword `kaizen-harness`. `npm search kaizen-harness`.
- `kaizen install <name>` resolves `kaizen-harness-<name>` from npm.
- Plugin installation shells out to `npm install -g` (npm must be available for plugin
  management; the kaizen binary itself requires no npm at runtime).
  Phase 3: bundle minimal npm client to remove this dependency.

**Release process:**
- Tag `v*` → GitHub Actions builds all 5 platform binaries + publishes to npm.
  Manual release in Phase 2.

## Known Limitations (Phase 2)

- **Plugin isolation:** Plugins run in-process and have full Node.js/Bun access.
  Only install plugins you trust. Sandboxing via worker_threads is Phase 3.
- **Single provider per session:** `ctx.runtime.llm` uses one LLM provider. Multi-provider
  sessions (e.g., routing different tool calls to different models) are Phase 3.
- **No dependency declarations between plugins:** `depends[]` resolves roles, not
  version ranges. If `kaizen-plugin-cli` requires a lifecycle feature added in
  lifecycle v2.0, there is no way to express that in Phase 2. Version negotiation is
  Phase 3.
- **Plugin resolution requires npm:** `kaizen install` and `kaizen plugin install` shell
  out to `npm`. Users running the compiled binary still need npm for plugin management.
  Bundled npm client is Phase 3.
